# 代码修复规则 - 禁止创建文档

## 核心原则

在修复代码、调试问题、优化性能或进行任何代码改动后：

- ❌ **绝对不要**创建任何 Markdown 文档来总结修复内容
- ❌ **绝对不要**创建 CHANGELOG、FIX_SUMMARY、UPDATE_LOG 等文件
- ❌ **绝对不要**创建任何形式的说明文档、修复报告、变更记录
- ✅ **只需要**在对话中简短说明做了什么改动即可

## 具体禁止行为

禁止创建以下类型的文件：
- `FIX_SUMMARY.md`
- `CHANGES.md`
- `UPDATE_LOG.md`
- `BUGFIX_REPORT.md`
- `CODE_CHANGES.md`
- 任何其他总结性、记录性的文档

## 正确做法

修复代码后，只需要：
1. 修改相关的代码文件
2. 在对话中用 1-2 句话简单说明改动
3. 完成，不创建任何额外文件

## 示例

❌ 错误做法：
```
修复了 bug，创建了 FIX_SUMMARY.md 文档记录修复内容...
```

✅ 正确做法：
```
已修复 DataList.jsx 中的分页问题，调整了 API 调用逻辑。
```

## 例外情况

唯一可以创建文档的情况：
- 用户**明确要求**创建文档
- 用户说"请写一个文档"、"帮我记录一下"等明确指令

除此之外，任何情况下都不要主动创建文档。

## 代码质量检查

修复代码后，必须执行以下检查：

1. **文件行数检查**
   - 统计修改后的文件总行数
   - 如果超过 1000 行，必须提示用户
   - 分析文件结构，建议如何拆分

2. **提示格式**
   ```
   ⚠️ 代码质量提醒：
   - 文件 xxx.js 有 XXXX 行，超过 1000 行建议值
   - 建议拆分方向：
     * 将 XXX 功能提取到独立模块
     * 将 YYY 工具函数移到 utils
     * 考虑按职责拆分为多个文件
   ```

3. **何时检查**
   - 每次修复、优化、添加功能后
   - 在完成修改、准备推送前
   - 不需要用户明确要求，自动执行
